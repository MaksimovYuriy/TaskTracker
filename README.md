# Task Tracker API

API-модуль трекера рабочих задач для медицинской информационной системы (МИС). Через модуль врачи и администраторы ставят задачи: операции, обходы, отчёты, звонки и т.п. Поддерживается периодичность задач (ежедневные, ежемесячные, на конкретные даты, чёт/нечет дни месяца) и теги.

## Стек

- **Ruby** 3.4.1
- **Rails** 7.1 (API-only)
- **PostgreSQL** 16+
- **Redis** + **Sidekiq** (фоновые задачи, cron-расписание через `sidekiq-cron`)
- **JSON:API** через `jsonapi-serializer`
- **Pagy** + **Ransack** — пагинация и фильтрация
- **rswag** — OpenAPI/Swagger из request-spec'ов

## Запуск

Два способа на выбор: нативно или через Docker Compose.

### Нативный запуск (требует локальные PostgreSQL 16+, Redis, Ruby 3.4.1)

```bash
bundle install
bin/rails db:setup              # create + migrate + seed (системные теги)
bin/rails s                     # API на http://localhost:3000
bundle exec sidekiq             # воркер с cron-задачей материализации
```

### Через Docker Compose (быстрый старт)

Поднимает 4 сервиса: `db` (Postgres 16), `redis`, `web` (Rails), `worker` (Sidekiq).

```bash
docker compose up --build       # на первом запуске собирается dev-образ Ruby
```

После старта:
- API: `http://localhost:3000`
- OpenAPI UI: `http://localhost:3000/api-docs`
- Sidekiq UI: `http://localhost:3000/sidekiq`

**Важно:** хост-порты 5432 / 6379 / 3000 должны быть свободны. Если у вас в WSL уже запущены нативные PostgreSQL/Redis — остановите их (`sudo service postgresql stop`, `sudo service redis-server stop`) или временно поправьте порт-маппинги в `docker-compose.yml`.

### Неописанные моменты и решение основных задач при разработке периодичности

Для настройки периодичности у задач была выбрана следующая схема:

- Если передается настройка периода, создается объект шаблона
- Из шаблона создаются объекты реальных задач

Такой принцип позволит избежать проблем с состоянием только конкретного экземпляра, так как обновлять статус или переносить время врач будет у реального объекта, а не у всей серии.

Для проблемы бесконечности объектов выбран способ, при котором объекты создаются раз в месяц, при этом если была создана задача с периодичностью в середине месяца, автоматически создаются экземпляры задач до конца текущего месяца.

Если ежемесячная джоба не создала ни одной задачи по шаблону - шаблон удаляется.

Так же предусмотрел возможность отменить весь шаблон через отдельный эндпоинт, по которому будет удален сам объект шаблона, а также отдельные экземпляры задач, поставленные позже текущего момента.
Также была неопределенность с валидацией задач и дублированием. Было принято решение на данном этапе обойтись обычным атрибутом :user_id у моделей Task и TaskTemplate, который позволил сделать валидацию по паре user_id - scheduled_at, один врач - одна задача на конкретное время.

Соотвественно можно задать тэги на шаблон, которые будут скопированы для реальных задач при инициализации экземпляров.
